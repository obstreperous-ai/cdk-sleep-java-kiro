# File extension validation via Step Functions Choice state (v2)

This change adds a `ValidateFileExtension` Choice state to the Step Functions pipeline, checking uploaded files for `.wav`, `.mp3`, or `.ogg` extensions before invoking any compute. A `ValidationError` Pass state injects synthetic error context on the otherwise path so the shared failure chain (`UpdateMetadataStatusFailed` → `PublishFailureNotification`) resolves its JsonPath references correctly. Defense-in-depth validation in the Lambda rejects unsupported formats even if the Choice state is bypassed. Two new test classes and fixes to four pre-existing `LambdaFunctionTest` failures round out the change.

**Watch for:** The Polly `iamResources` remains `List.of("*")` and the corresponding test intentionally omits a Resource scope assertion (confirmed, unchanged from v1). A handful of `InputValidationTest` methods still rely on substring presence rather than structural JSON parsing, making them fragile to refactoring (likely, low severity since the stronger tests in `PipelineWiringTest` cover the same ground).

**Verdict**: APPROVED

## High-level view

The critical v1 issue — the Choice `otherwise` path referencing unpopulated `$.error` fields — is resolved by a `ValidationError` Pass state that injects `{"Error": "ValidationError", "Cause": "Unsupported file extension"}` at `$.error` before entering the shared failure chain. Three new structural tests parse the actual state machine definition JSON and assert on `Default`/`Next` fields, making accidental breakage highly visible.

The remaining `InputValidationTest` methods use substring checks against the entire template JSON. They add no regression safety beyond what the structural tests already provide, but are low-maintenance and serve as readable specification.

The case sensitivity asymmetry between the Choice state (`StringMatches` is case-sensitive) and the Lambda (`lower().endswith(...)`) means a file like `track.WAV` would be rejected at the state machine level but would pass the Lambda's check. Benign in the current architecture since the Choice gates first, but makes the "defense-in-depth" claim technically incomplete.

## ValidationError Pass state implementation

The Pass state at lines 269–275 of `CdkBaseStack.java`:

```java
Pass validationErrorState = Pass.Builder.create(this, "ValidationError")
        .result(Result.fromObject(Map.of(
                "Error", "ValidationError",
                "Cause", "Unsupported file extension"
        )))
        .resultPath("$.error")
        .build();
```

The wiring chain — `ValidateFileExtension.Default → ValidationError → UpdateMetadataStatusFailed → PublishFailureNotification` — is verified by three structural tests that parse the DefinitionString JSON and assert on `Default`/`Next` field values directly:

```
PutMetadataRecord.Next == "ValidateFileExtension"
ValidateFileExtension.Default == "ValidationError"
ValidationError.Next == "UpdateMetadataStatusFailed"
UpdateMetadataStatusFailed.Next == "PublishFailureNotification"
```

## Case sensitivity asymmetry between validation layers

The Step Functions `StringMatches` condition is case-sensitive per the ASL spec. A file `recording.WAV` would not match `*.wav` and would hit the `otherwise` path, receiving a `FAILED` status and failure notification. Meanwhile, the Lambda's `object_key.lower().endswith(valid_extensions)` would accept it. If a future change removes or reorders the Choice state, the Lambda would silently accept uppercase extensions that were previously rejected, producing inconsistent behavior depending on which validation layer runs first.
