# File extension validation via Step Functions Choice state

This change introduces a `ValidateFileExtension` Choice state into the Step Functions pipeline that checks uploaded files for `.wav`, `.mp3`, or `.ogg` extensions before processing begins. It also adds defense-in-depth validation in the Lambda function, fixes four pre-existing test failures in `LambdaFunctionTest`, and introduces two new test classes (`InputValidationTest`, `PipelineWiringTest`) covering the validation logic and end-to-end wiring. Documentation is updated to reflect the new pipeline topology.

**Watch for:** The Choice state's `otherwise` path routes directly to `UpdateMetadataStatusFailed`, which references `$.error.Cause` and `$.error.Error` via JsonPath — fields that only exist when a Catch block populates them (confirmed, lines 229 and 248-249 of `CdkBaseStack.java`). Invalid-extension uploads will hit a runtime failure in the error-handling path itself. Several tests in `PipelineWiringTest` assert only that a string exists somewhere in the synthesized template JSON, making them unable to catch regressions or mis-wiring.

## High-level view

The Choice state is inserted between `PutMetadataRecord` and `ProcessAudioMetadata`, branching on `StringMatches` against `$.object.key`. Three `.when()` clauses handle valid extensions and an `.otherwise()` routes to the existing failure chain. The design is clean — validation runs before any compute — but it reuses a failure path that assumes error context from a Catch block, creating a broken path for the exact scenario the Choice state is meant to handle.

The Lambda's new file-extension check provides genuine defense-in-depth: if the Choice state were ever bypassed (e.g., a direct `StartExecution` call with a crafted payload), the Lambda would still reject unsupported formats. The Step Functions `StringMatches` is case-sensitive while the Lambda uses `lower()`, so uppercase extensions like `.WAV` would be rejected by the Choice state but accepted by the Lambda — a benign asymmetry since the Choice state gates first.

The new test classes provide breadth but many individual tests are structurally weak, asserting only substring presence in a large JSON blob rather than verifying structural relationships between states.

## Choice state runtime failure on the `otherwise` path

The `UpdateMetadataStatusFailed` task writes `$.error.Cause` into DynamoDB as the `errorInfo` attribute:

```java
":errorInfo", Map.of("S", JsonPath.stringAt("$.error.Cause"))
```

And `PublishFailureNotification` publishes:

```java
"error", JsonPath.stringAt("$.error.Error"),
"cause", JsonPath.stringAt("$.error.Cause")
```

When a Catch block fires, it places `{"Error": "...", "Cause": "..."}` at `$.error` (per the `resultPath("$.error")` config). But when the Choice state's `otherwise` transitions directly to `UpdateMetadataStatusFailed`, the state input is the unmodified output of `PutMetadataRecord` — it has no `$.error` field. Step Functions will throw a `States.Runtime` error because the JsonPath reference cannot be resolved, and since there is no Catch on the failure path itself, the execution will abort with a terminal failure.

The exact scenario being validated — an upload with an unsupported extension — will produce a broken execution rather than a clean FAILED record and notification. The fix is to insert a Pass state that injects a synthetic error payload before entering the shared failure path, or to build a parallel failure branch for the validation case that uses static strings instead of JsonPath references to error fields.

## Test coverage gaps in wiring assertions

`PipelineWiringTest` uses `template.toJSON().toString()` and `assertTrue(json.contains("StateName"))` for most assertions. This technique verifies that a state name appears *somewhere* in the synthesized CloudFormation template — which includes resource logical IDs, policy documents, and definition strings — but does not verify structural relationships (which state transitions to which). `testValidationFailureRoutesToUpdateMetadataStatusFailed` would pass even if the Choice state's `Default` pointed to a completely different state, as long as `UpdateMetadataStatusFailed` existed anywhere in the template.

`InputValidationTest.testChoiceStatePositionedAfterPutMetadataRecord` checks that both state names exist and that `PutMetadataRecord` appears at a lower string index than `ValidateFileExtension` — a position check on an unordered JSON map that could pass or fail depending on serialization order rather than actual wiring.

To make these tests meaningful, parse the state machine `DefinitionString` from the CloudFormation template, deserialize it as JSON, and assert on the `States.ValidateFileExtension.Default` field value and `States.PutMetadataRecord.Next` field value directly.
