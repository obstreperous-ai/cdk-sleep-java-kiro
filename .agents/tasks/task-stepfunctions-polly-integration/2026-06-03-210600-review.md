# Step Functions state machine with Polly SynthesizeSpeech skeleton

A Step Functions state machine is added to the CDK stack, containing a single `CallAwsService` task that invokes `polly:synthesizeSpeech` with placeholder parameters. The EventBridge rule's target is swapped from a CloudWatch LogGroup (placeholder) to an `SfnStateMachine` target that passes `$.detail` as input. CloudWatch logging (ALL level) is enabled on the state machine via a dedicated log group. Five new CDK assertion tests cover the new resources.

Watch for: The IAM policy grants `polly:synthesizeSpeech` and all CloudWatch Logs delivery permissions on `Resource: "*"` in a single combined statement (confirmed). The Polly task's response is a binary audio stream that Step Functions cannot meaningfully store in `$.pollyResult` at runtime (confirmed). The test for the Polly task only asserts `DefinitionString` exists, not that it references Polly (confirmed).

## High-level view

The state machine is a one-state skeleton: Start -> SynthesizeSpeech -> End. It uses `CallAwsService` (the SDK integration pattern) to call Polly directly without a Lambda intermediary. The `iamResources(List.of("*"))` means the auto-generated policy scopes to all resources—should be tightened when real audio processing is added.

CDK automatically creates a separate `events.amazonaws.com` role with `states:StartExecution` permission scoped to the state machine ARN. The rule target uses `InputPath: "$.detail"`, so the state machine receives the raw S3 event detail object (`bucket`, `object`, `request-id`, etc.) as execution input. The Polly task ignores this input (static parameters), but the shape is ready for a future state that reads the object key.

The Polly `SynthesizeSpeech` SDK call returns an `AudioStream` (binary) plus metadata fields. Step Functions cannot store binary data in state JSON, so the current skeleton will succeed at synth/deploy time but will produce a runtime error or empty result when actually executed. The limitation should be documented or addressed before wiring real input—`StartSpeechSynthesisTask` (async, writes to S3) is the appropriate API for Step Functions orchestration.

The test suite validates resource existence and IAM permissions but avoids asserting on the state machine definition content. The tests would pass even if the Polly task were replaced with a different service call, reducing regression-detection value.

## Polly SDK integration via CallAwsService

The `CallAwsService` construct generates a Task state with resource ARN `arn:{partition}:states:::aws-sdk:polly:synthesizeSpeech`. The parameters are static placeholders (`Text`, `VoiceId: "Joanna"`, `OutputFormat: "mp3"`). At runtime, Step Functions calls the Polly API and attempts to place the response at `$.pollyResult`.

The Polly `SynthesizeSpeech` API response includes `AudioStream` (a blob), `ContentType`, and `RequestCharacters`. The SDK integration in Step Functions can handle scalar response fields but cannot deserialize a binary stream into the state JSON. In practice, the execution will either error or produce a truncated/empty result.

For a production pipeline, `StartSpeechSynthesisTask` (the asynchronous API that writes output to S3) is the appropriate Polly API for Step Functions orchestration. The ARCHITECTURE.md Orchestration Layer section doesn't call out this limitation.

## IAM policy structure

CDK merges the logging permissions and the Polly permission into a single policy statement:

```json
{
  "Action": [
    "logs:CreateLogDelivery",
    "logs:DeleteLogDelivery",
    "logs:DescribeLogGroups",
    "logs:DescribeResourcePolicies",
    "logs:GetLogDelivery",
    "logs:ListLogDeliveries",
    "logs:PutResourcePolicy",
    "logs:UpdateLogDelivery",
    "polly:synthesizeSpeech"
  ],
  "Effect": "Allow",
  "Resource": "*"
}
```

All actions share `Resource: "*"`. For Polly this is unavoidable (no resource-level permissions). For the `logs:*` actions, CDK generates this pattern because Step Functions log delivery requires cross-log-group operations—a CDK default, not a code issue.

## Test coverage gaps

`testStateMachineDefinitionContainsPollyTask` asserts only that `DefinitionString` is present (not null). Since `DefinitionString` is a `Fn::Join` intrinsic in CloudFormation, a naive `Match.serializedJson` approach won't work directly, but the CDK assertions library supports `Match.stringLikeRegexp` which could verify the definition contains `polly` or `synthesizeSpeech`. As written, this test would pass if the definition body were changed to any other service call.

`testEventBridgeRuleTargetsStateMachine` asserts that the target Arn contains a `Ref` but doesn't verify it references the state machine specifically, nor does it assert on `InputPath` or `RoleArn`.

Not tested: The `InputPath: "$.detail"` on the EventBridge target. The log group retention and removal policy. The `resultPath` configuration on the Polly task.
