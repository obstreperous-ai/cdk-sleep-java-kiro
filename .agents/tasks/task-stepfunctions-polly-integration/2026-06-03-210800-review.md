# Step Functions state machine with Polly SynthesizeSpeech skeleton (v2)

A Step Functions state machine is wired into the existing EventBridge rule, replacing the CloudWatch LogGroup placeholder target. The machine contains a single `CallAwsService` task calling `polly:synthesizeSpeech` with static placeholder parameters. CloudWatch logging at ALL level is attached, and five CDK assertion tests validate the new infrastructure. ARCHITECTURE.md documents both the implementation and the known synchronous-Polly limitation.

Watch for: The EventBridge rule target test asserts only that a `Ref`-based ARN exists in the Targets array, not that it references the state machine specifically — any resource with a `Ref` in that position would pass (confirmed). The `InputPath` on the EventBridge target is not tested (confirmed).

**Verdict**: APPROVED

## High-level view

The v1 review raised three issues. The Polly runtime limitation (#1) is now explicitly documented in ARCHITECTURE.md with a clear note explaining why `SynthesizeSpeech` is a placeholder and what replaces it. The Polly test (#2) now uses `template.toJSON().toString()` with `assertTrue(contains("polly"))` and `assertTrue(contains("synthesizeSpeech"))`, which catches removal of the Polly integration from the definition. The IAM wildcard (#3) remains as a CDK-generated default — no action needed for a skeleton.

The remaining gaps are minor: the EventBridge target test is structurally weak (it would pass if the target ARN happened to reference any resource via `Ref`, not specifically the state machine), and the `InputPath: "$.detail"` configuration is untested. Neither is blocking for a skeleton milestone, but both reduce regression-detection confidence if the wiring changes in a future iteration.

## EventBridge target test specificity

The synthesized template shows the rule target as:

```json
{
  "Arn": { "Ref": "SleepAudioPipelineStateMachine741D1700" },
  "InputPath": "$.detail",
  "RoleArn": { "Fn::GetAtt": ["SleepAudioPipelineStateMachineEventsRole0816CD2D", "Arn"] }
}
```

The test asserts:

```java
Match.objectLike(Map.of(
    "Arn", Match.objectLike(Map.of(
        "Ref", Match.anyValue()
    ))
))
```

`Match.anyValue()` means any logical ID satisfies the assertion. If a future refactor introduces a second state machine or swaps the target to a different resource, this test would still pass. Using a `Match.stringLikeRegexp(".*StateMachine.*")` on the `Ref` value—or asserting on both `Arn` and `RoleArn` together—would pin the test to the intended target.

The `InputPath: "$.detail"` field is the contract between EventBridge and the state machine's expected input shape. It appears in the template but no test covers it. Adding `"InputPath", "$.detail"` to the `Match.objectLike` map would lock this contract.

Neither gap is blocking for a placeholder skeleton, but both should be addressed when the state machine starts consuming the event input (reading `$.bucket.name` or `$.object.key` in a future Validate state).
