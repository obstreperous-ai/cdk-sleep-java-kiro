# Pipeline testing expansion and deployment prep skeleton

This change adds 19 new tests across four new test files (end-to-end flow verification, resource count snapshots, multi-environment context, pipeline construct synthesis), introduces a CDK Pipelines `PipelineStack` skeleton for future CI/CD, and wires environment context into `CdkBaseStack` with tag propagation. The implementation follows TDD: tests were written first, then production code was added to make them pass.

**Watch for:** The `PipelineStack` embeds a hardcoded placeholder CodeStar connection ARN and account ID that will deploy real AWS resources if the stack is ever synthesized in a pipeline context (confirmed). The `CdkBaseStack` context read uses an unchecked cast from `Object` to `String` that will throw `ClassCastException` if the context value is not a string (confirmed). The `PipelineStack` is never instantiated in `CdkBaseApp`, so it's inert today, but its test exercises synthesis—meaning it could silently drift from the main stack without CI catching it.

## High-level view

The test suite grows from 53 to 72 tests. `EndToEndFlowTest` is the most valuable addition—it parses the state machine's `DefinitionString` JSON and walks the `Next` pointers sequentially, verifying the happy-path chain and the error-path chain independently. This catches state-ordering regressions that the existing `PipelineWiringTest` (which only checks state existence via string search) would miss. `SnapshotTest` pins resource counts, providing a coarse guard against accidental resource additions or removals.

Multi-environment support is minimal: `CdkBaseStack` reads a string context value called `environment`, defaults to `"dev"`, and applies it as a tag. No behavioral branching per environment exists yet—both `dev` and `prod` produce identical infrastructure aside from the tag value. The tests assert only that the mechanism works, not that it drives meaningful differentiation.

`PipelineStack` is a standalone skeleton that creates a `CodePipeline` with a connection source and shell synth step. It is not wired into `CdkBaseApp` and cannot deploy anything today. The placeholder connection ARN (`arn:aws:codestar-connections:us-east-1:123456789012:connection/placeholder-connection-id`) and repo reference (`owner/repo`) are the obvious activation blockers. The CI workflow now runs `cdk synth -c environment=prod` as a second synthesis step, but neither synth step exercises `PipelineStack`.

<summary>Issues (4)</summary>

1. **Unsafe context cast** — `this.getNode().tryGetContext("environment")` returns `Object`; casting to `String` without an `instanceof` check will throw `ClassCastException` if cdk.json or `-c` supplies a non-string value (e.g., a number or boolean). Add an `instanceof String` guard or use `Objects.toString()`.
2. **Placeholder credentials in source** — The hardcoded ARN `arn:aws:codestar-connections:us-east-1:123456789012:connection/placeholder-connection-id` and repository `owner/repo` are functional enough to deploy a broken pipeline if someone adds `new PipelineStack()` to the app. Consider extracting these to context values that fail loudly when unset.
3. **PipelineStack is untested by CI synth** — The CI workflow runs `cdk synth` which only synthesizes stacks in `CdkBaseApp`. Since `PipelineStack` is never instantiated there, CI doesn't validate it. A CI-level regression (e.g., a dependency version bump breaking the pipeline construct) would be caught only by the unit test, not by the full synth step.
4. **Snapshot test fragility** — `SnapshotTest` pins exact resource counts (S3=2, DDB=1, etc.). CDK version bumps that add custom-resource Lambda functions or log groups will break these tests without any real regression. Consider using `>=` assertions for resources CDK may auto-generate (Lambda, LogGroup) consistently, not just for Lambda.

## Unsafe context cast in CdkBaseStack

The environment-reading code:

```java
String environment = (String) this.getNode().tryGetContext("environment");
```

`tryGetContext` returns `Object`. If a user passes `-c environment=true` or if some automation passes a non-string JSON value, this line throws `ClassCastException` at synthesis time with no useful error message. The null check that follows handles the absent case but not the wrong-type case. A defensive pattern:

```java
Object envObj = this.getNode().tryGetContext("environment");
String environment = envObj instanceof String ? (String) envObj : "dev";
```

## PipelineStack isolation and placeholder risk

`PipelineStack` embeds:

```java
CodePipelineSource.connection("owner/repo", "main",
    ConnectionSourceOptions.builder()
        .connectionArn("arn:aws:codestar-connections:us-east-1:123456789012:connection/placeholder-connection-id")
        .build())
```

When someone activates the pipeline, they'll need to replace two separate string literals buried in the constructor. If either is missed, CloudFormation will attempt to create a pipeline pointing at `owner/repo` with a non-existent connection. Extracting these to required context values (failing synthesis if absent) would make activation safer and the failure mode obvious.

## End-to-end flow test gap: new terminal states

The `testAllCatchBlocksRouteToErrorHandler` method iterates every state and checks that all `Catch` clauses route to `UpdateMetadataStatusFailed`. It excludes states whose names start with "UpdateMetadataStatus" or "Publish" (the error-path states themselves) and states with `Type == "Choice"` or `Type == "Succeed"`. This means if a new task state is added to the error path or as a terminal branch that doesn't start with one of those prefixes, and it lacks a Catch clause, this test won't flag it. The coverage is good for the current state machine shape but won't automatically extend to new states outside the naming convention.

## Snapshot test fragility surface

`SnapshotTest` asserts exact counts: S3=2, DDB=1, SFN=1, EventBridge=1, SNS=2, KMS=1, Lambda≥1. The `>=1` for Lambda acknowledges CDK's custom-resource handlers, but the same reasoning applies to KMS keys (CDK may generate additional keys for pipeline encryption or log encryption in future constructs). If a CDK version bump adds an extra KMS key, the test will fail spuriously. Documenting which counts are "exact by design" vs. "exact because CDK currently generates this many" in comments would help maintainers distinguish real failures from CDK-upgrade noise.
