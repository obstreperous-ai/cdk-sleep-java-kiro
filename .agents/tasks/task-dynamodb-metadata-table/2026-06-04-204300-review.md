# DynamoDB metadata table and state machine integration

This change adds a DynamoDB table (`SleepAudioMetadataTable`) to the CDK stack and extends the Step Functions state machine from a single Polly task into a three-step chain: PutItem (writes a PROCESSING record) â†’ Polly SynthesizeSpeech â†’ UpdateItem (sets COMPLETED). The table uses on-demand billing, point-in-time recovery, AWS-owned encryption, and RETAIN removal policy. IAM is handled via `grantReadWriteData` on the state machine. Eight CDK assertion tests cover the table properties and state machine integration.

**Watch for:** Dual IAM grant path â€” `CallAwsService.iamResources(List.of("*"))` generates its own `Allow *` statements alongside the scoped policy from `grantReadWriteData`, resulting in over-broad permissions (confirmed). The UpdateItem task reads `$.object.key` from the execution state, but the preceding Polly task writes its result to `$.pollyResult` which may clobber or restructure the state â€” the S3 event fields must survive the Polly step untouched for UpdateItem to reference them correctly (likely). No error handling exists in the chain; a DynamoDB failure leaves the record permanently in PROCESSING with no retry or catch (confirmed).

## High-level view

The state machine chain introduces a bookkeeping pattern where the pipeline records its own progress: write PROCESSING before work begins, write COMPLETED after work finishes. Both DynamoDB tasks use `CallAwsService` with direct SDK-level parameters (DynamoDB JSON wire format with explicit type descriptors).

The IAM story has a gap: each `CallAwsService` task declares `iamResources(List.of("*"))` which CDK translates into an IAM statement with `Resource: *`. The subsequent `grantReadWriteData` call adds a properly-scoped policy, but it doesn't remove the wildcard statements. The deployed role ends up with both â€” the scoped grant is effectively redundant.

The chain has no error states. If PutItem fails (throttle, validation error, conditional check), the entire execution fails without catching or retrying. If Polly succeeds but UpdateItem fails, the record is stuck at PROCESSING forever with no mechanism to reconcile.

## Dual IAM grant and wildcard resources

Each `CallAwsService` task in CDK auto-generates an IAM policy statement from `iamResources`. The code passes `List.of("*")`:

```java
.iamResources(List.of("*"))
```

This produces statements like `{ "Action": "dynamodb:PutItem", "Resource": "*" }`. Meanwhile, `metadataTable.grantReadWriteData(stateMachine)` adds a second policy with the table ARN as the resource. The net effect is that the state machine role can `PutItem` and `UpdateItem` against any DynamoDB table in the account. The fix is straightforward â€” pass `List.of(metadataTable.getTableArn())` to both `CallAwsService` builders.

## State data flow across the chain

The execution input arrives from EventBridge containing `$.bucket.name` and `$.object.key`. Both PutItem and Polly use `resultPath` (not `outputPath`), so the original input fields survive through the chain today. But this is fragile â€” any future task that sets `outputPath` or uses the default result path (`$`) would wipe the S3 event data. A defensive pattern would be to nest the original input under a stable key at the start of the chain, or use `inputPath` on the DynamoDB tasks to be explicit about what they consume.

## Missing error handling and stuck-in-PROCESSING records

The chain is purely linear with no retry or catch configuration:

```
PutItem â†’ Polly â†’ UpdateItem
```

On-demand DynamoDB tables can still throttle during partition splits. If Polly succeeds but UpdateItem fails, the metadata record shows PROCESSING permanently. There's no dead-letter queue, no TTL-based cleanup, and no reconciliation mechanism.

At minimum, `addRetry` with `errors(List.of("DynamoDB.DynamoDbException"))` and an exponential backoff would handle transient failures. For the stuck-PROCESSING scenario, a `addCatch` on UpdateItem that writes a FAILED status would at least make the state queryable.

## Test coverage gaps

The IAM test asserts that `dynamodb:PutItem` and `dynamodb:UpdateItem` appear in an Allow statement but doesn't assert the resource is scoped â€” which currently would pass even with the wildcard `*` resource.

Not tested: the ordering of states in the chain (PutItem before Polly before UpdateItem), the actual parameter values passed to DynamoDB (table name reference, item schema, update expression), or the `resultPath` configuration that keeps state flowing between steps.
