# DynamoDB metadata table and state machine integration (v2)

This change adds a DynamoDB table (`SleepAudioMetadataTable`) to the CDK stack and extends the Step Functions state machine into a three-step chain: PutItem (PROCESSING) → Polly SynthesizeSpeech → UpdateItem (COMPLETED). The table uses on-demand billing, point-in-time recovery, AWS-owned encryption, and RETAIN removal policy. DynamoDB `CallAwsService` tasks now scope their `iamResources` to the table ARN, and both carry `addRetry` with `States.ALL` / maxAttempts=3 / backoffRate=2.0. Eight CDK assertion tests cover table properties and state machine integration.

**Watch for:** The retry configuration catches `States.ALL` which includes non-transient errors like `DynamoDB.ValidationException` — a conditional-check failure or malformed request will be retried three times before failing, wasting time on errors that can never succeed (confirmed). The IAM permission test doesn't assert the resource is scoped to the table ARN — it would pass even if `iamResources` were still `"*"` (confirmed). Redundant IAM policy statements from both `iamResources` and `grantReadWriteData` targeting the same ARN; not a security issue but adds policy noise (confirmed).

## High-level view

The v1 wildcard IAM concern is resolved — both `CallAwsService` tasks pass `List.of(metadataTable.getTableArn())` and the synthesized template confirms scoping via `Fn::GetAtt`. The role still carries two DynamoDB-related statements (one from `CallAwsService`, one from `grantReadWriteData`) which is redundant but not dangerous since `iamResources` is a required parameter that can't be omitted.

The retry strategy addresses the v1 error-handling concern but uses `States.ALL` which doesn't distinguish transient failures (throttling) from permanent ones (validation errors). A malformed item will be retried three times before surfacing the error, and there's no `addCatch` to handle exhausted retries — a failed UpdateItem leaves the record stuck in PROCESSING.

The test suite covers table configuration thoroughly but the IAM assertion checks actions and effect without asserting the `Resource` field, so it cannot detect a regression to wildcard permissions.

## Retry breadth on DynamoDB tasks

Both `putItemTask` and `updateItemTask` carry:

```java
putItemTask.addRetry(RetryProps.builder()
        .errors(List.of("States.ALL"))
        .maxAttempts(3)
        .backoffRate(2.0)
        .build());
```

`States.ALL` matches every error including `DynamoDB.ValidationException` (schema violation, missing required attribute) and `DynamoDB.ConditionalCheckFailedException`. These are deterministic failures — repeating the same request will produce the same error. The retry adds latency (up to ~7 seconds with backoff) before the execution ultimately fails.

For the PutItem task, if a duplicate `audioId` triggers a conditional check or a malformed item fails validation, the pipeline wastes three attempts. Narrowing to transient errors (`DynamoDB.ProvisionedThroughputExceededException`, `DynamoDB.InternalServerError`, `DynamoDB.ItemCollectionSizeLimitExceededException`, `States.Timeout`) would preserve the retry benefit for throttling without masking bugs.

There's no `addCatch` on either DynamoDB task. If retries are exhausted, the entire execution fails and the record may be stuck in PROCESSING with no mechanism to reconcile. A catch transitioning to a failure state would make the failure mode observable without requiring CloudWatch log analysis.

## IAM permission test gap

The permission test asserts:

```java
template.hasResourceProperties("AWS::IAM::Policy", Match.objectLike(Map.of(
    "PolicyDocument", Match.objectLike(Map.of(
        "Statement", Match.arrayWith(List.of(
            Match.objectLike(Map.of(
                "Action", Match.arrayWith(List.of(
                    "dynamodb:PutItem",
                    "dynamodb:UpdateItem"
                )),
                "Effect", "Allow"
            ))
        ))
    ))
)));
```

This would pass even if `iamResources` were reverted to `"*"` because the test doesn't inspect the `Resource` field. The scoped `CallAwsService` statement uses lowercase actions (`dynamodb:putItem`) while the `grantReadWriteData` statement uses PascalCase (`dynamodb:PutItem`) — so the test is actually matching the broader grant statement, not the scoped one from `CallAwsService`. Adding a `Resource` assertion matching `Fn::GetAtt` of the table would guard against regressions to wildcard permissions.

## Dual IAM grant producing redundant statements

The synthesized template shows two DynamoDB statements on the state machine role:

```
Statement 1 (from CallAwsService iamResources):
  Action: [dynamodb:putItem, dynamodb:updateItem]
  Resource: { Fn::GetAtt: [SleepAudioMetadataTable, Arn] }

Statement 2 (from grantReadWriteData):
  Action: [dynamodb:BatchGetItem, ..., dynamodb:PutItem, ..., dynamodb:UpdateItem, ...]
  Resource: { Fn::GetAtt: [SleepAudioMetadataTable, Arn] }
```

Statement 1 is a strict subset of Statement 2. The `iamResources` parameter is required by `CallAwsService` (won't compile without it), so the cleanest option is to remove `grantReadWriteData` and rely on the `CallAwsService`-generated permissions which already cover the exact actions used (putItem + updateItem). If future tasks need broader DynamoDB access (Query, Scan), the grant can be re-added then.
