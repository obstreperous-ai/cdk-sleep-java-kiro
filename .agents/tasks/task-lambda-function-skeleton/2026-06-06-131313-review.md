# SleepAudioProcessor Lambda skeleton and LambdaInvoke integration

A Python 3.12 Lambda function (`SleepAudioProcessor`) is added to validate audio metadata from S3 events, and a `LambdaInvoke` step is wired into the Step Functions chain between PutItem and Polly. The CDK stack grants the Lambda read-only DynamoDB access, the state machine gets `lambda:InvokeFunction`, and error handling routes Lambda failures through the existing Catch-to-failure-path pattern. Tests use CDK assertions to verify resource properties and IAM policies.

**Watch for:** The Lambda handler swallows all exceptions and returns a success response with `status: ERROR` instead of raising — meaning the Step Functions Catch block will never fire on validation failures (confirmed). The `payloadResponseOnly(false)` setting wraps the response in metadata, which means downstream states accessing `$.lambdaResult` get a nested structure including `Payload`, `SdkHttpMetadata`, etc., not the flat dict the handler returns (confirmed). Test assertions for DynamoDB read access may not match CDK's actual `grantReadData` action set (likely).

## High-level view

The Lambda pulls `bucket.name` and `object.key` from the incoming state, validates presence, and returns an enriched metadata dict. The core design issue is that validation errors are caught and returned as a successful Lambda invocation rather than raised as exceptions, so the Step Functions error-handling topology never activates for the one failure mode this function is designed to detect.

The `payloadResponseOnly(false)` default means the Lambda's response dict is nested under `$.lambdaResult.Payload`, which will matter when downstream tasks try to use the validated metadata. No downstream task references it yet, but the ARCHITECTURE.md implies consumption.

The test file relies on string-contains checks against serialized CloudFormation JSON for structural properties like chain ordering and Catch blocks. These tests will pass for any template that mentions "ProcessAudioMetadata" anywhere, not necessarily in the correct position. One test's name and comment ("basic execution permissions" / "CloudWatch Logs") contradict what it actually asserts (DynamoDB actions).

1. **Lambda swallows errors instead of raising** — The handler catches all exceptions and returns `{"status": "ERROR", ...}` as a successful invocation. Step Functions Catch blocks only trigger on Lambda invocation failures (raised exceptions), so validation errors silently propagate as success. ## Lambda error semantics vs. Step Functions Catch topology

The handler's try/except block on lines 34-61 of `index.py` catches `Exception`, logs it, and returns a dict with `"status": "ERROR"`. From Step Functions' perspective, the Lambda invocation succeeded (HTTP 200, no function error). The `addCatch` on `lambdaInvokeTask` with `States.ALL` only fires when the Lambda raises an unhandled exception (which surfaces as a Lambda function error in the Step Functions runtime). Since all exceptions are caught inside the handler, the Catch block is effectively dead code for application-level errors.

This creates a fail-open condition: if `bucket.name` is missing from the input, the pipeline continues to Polly with `$.lambdaResult` containing `{"status": "ERROR", "error": "Missing required field: bucket.name"}` rather than routing to the failure path. Polly will then fail on its own terms (unrelated error), or worse, proceed with stale state data.

The fix is either: (a) remove the try/except and let `ValueError` propagate as a Lambda function error that triggers the Catch, or (b) add a Choice state after the Lambda that inspects `$.lambdaResult.Payload.status` and routes ERROR to the failure path. Option (a) is simpler and matches the ARCHITECTURE.md description of Catch-based error handling.

## LambdaInvoke payload wrapping

`payloadResponseOnly(false)` means the Step Functions task uses the `Invoke` API directly rather than the optimized Lambda integration. The response stored at `$.lambdaResult` will be:

```json
{
  "ExecutedVersion": "$LATEST",
  "Payload": { "status": "VALIDATED", "audioId": "...", ... },
  "SdkHttpMetadata": { ... },
  "SdkResponseMetadata": { ... },
  "StatusCode": 200
}
```

The ARCHITECTURE.md states the Lambda "returns enriched metadata with status=VALIDATED, audioId, and processingTimestamp" which implies downstream consumption. When that happens, the path will be `$.lambdaResult.Payload.audioId`, not `$.lambdaResult.audioId`. Setting `payloadResponseOnly(true)` would flatten this and make the integration consistent with what the docs describe.

## Unused TABLE_NAME and grantReadData

The Lambda declares `TABLE_NAME = os.environ.get("TABLE_NAME", "")` at module level but never references it in the handler body. The CDK stack calls `metadataTable.grantReadData(audioProcessorFunction)`, which generates an IAM policy allowing DynamoDB read operations. Since the handler performs no DynamoDB reads, this is dead permission surface area that violates least-privilege in the current state.

## Test coverage gaps

The three tests that use `template.toJSON().toString()` with `contains()` are structurally weak:

- `testStateMachineDefinitionContainsLambdaInvoke` checks for `"lambda:invoke"` — this is an IAM action string that could match in an IAM policy rather than the state machine definition.
- `testStateMachineDefinitionContainsCatchForLambdaTask` checks for `"Catch"` — every Catch block in the state machine (there are four) will satisfy this. It doesn't verify the Catch is specifically on the Lambda task.
- `testStateMachineChainOrderPutItemThenLambdaThenPolly` checks for `"ProcessAudioMetadata"` — proves the task exists but not its position in the chain.

A stronger approach would use `Match.serializedJson` on the `DefinitionString` to verify structural ordering within the state machine definition.

The `testLambdaExecutionRoleHasBasicExecutionPermissions` test comment says "Lambda execution role should have CloudWatch Logs permissions" but actually asserts DynamoDB actions (`BatchGetItem`, `GetRecords`). The test name and comment are misleading — it's verifying the broader read-data grant, not CloudWatch Logs permissions.

## Diagram consistency

The Mermaid diagram now shows two parallel paths out of `SFN`: one to `ProcessAudio` and one to `Validate`. The `Validate` node is a planned (not implemented) component. The diagram implies both run in parallel (`SFN --> ProcessAudio` and `SFN --> Validate` as separate edges), but the actual implementation is sequential: PutItem -> ProcessAudioMetadata -> Polly. Having both without showing they're sequential alternatives (one implemented, one planned) is visually misleading about the actual execution topology.
