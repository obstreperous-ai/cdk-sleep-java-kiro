# SNS notifications and Catch-based error handling for the sleep audio pipeline

This change adds two KMS-encrypted SNS topics (Completed/Failed) and a Catch block on the Polly task to route errors into a failure path that updates DynamoDB with FAILED status and publishes to the failure topic. The happy path now extends through a success SNS publish after the COMPLETED update. Tests use CDK assertions and JSON string-contains checks.

**Watch for:** Catch coverage is limited to the Polly task only — failures in the DynamoDB UpdateItem (COMPLETED) or SNS success publish steps are unhandled (confirmed). The SNS publish tasks send the entire state object (`$`) as the message payload, which leaks internal pipeline state (putItemResult, pollyResult, raw error objects) to any subscriber (confirmed). Several tests rely on string-contains checks against the full template JSON, which makes them brittle and non-specific (confirmed).

## High-level view

The error handling architecture follows a sensible pattern: Catch on the task most likely to fail (Polly), route to a DynamoDB status update, then notify. But the Catch only covers one step in a four-step chain. If `updateItemTask` (COMPLETED) or `successPublishTask` fails, the execution fails without updating DynamoDB to FAILED or notifying anyone. The pipeline lacks a top-level error handler.

SNS messages are constructed by forwarding the entire Step Functions state (`$`) to the topic. This means subscribers receive internal state (DynamoDB PutItem responses, Polly result objects, raw error structures) rather than a curated notification payload. This leaks implementation details and makes downstream consumers coupled to internal state structure.

The test suite confirms the constructs exist and exercises IAM assertions properly, but three of the six tests use `template.toJSON().toString().contains(...)` which matches anywhere in the CloudFormation template — not specifically within the state machine definition.

## Catch scope and unhandled failure modes

The Catch block is applied only to `pollyTask`:

```java
pollyTask.addCatch(failureUpdateTask, CatchProps.builder()
        .errors(List.of("States.ALL"))
        .resultPath("$.error")
        .build());
```

The chain is `PutItem → Polly → UpdateItem(COMPLETED) → SnsPublish(success)`. If `UpdateItem(COMPLETED)` throws a non-retryable error (after retry exhaustion), or if `successPublishTask` fails, the execution terminates with a FAILED status in Step Functions but the DynamoDB record remains in COMPLETED or PROCESSING state — no failure notification is sent.

`putItemTask` also has no Catch. If the initial metadata record fails to write beyond retries, the execution fails without any external signal (no record exists to update, no notification sent). This might be acceptable at the infrastructure level, but it means the pipeline can drop work silently.

## SNS message payload leaks internal state

Both publish tasks use `TaskInput.fromJsonPathAt("$")` as the message body:

```java
SnsPublish successPublishTask = SnsPublish.Builder.create(this, "PublishSuccessNotification")
        .topic(completedTopic)
        .message(TaskInput.fromJsonPathAt("$"))
        ...
```

After the happy path completes, `$` contains `{object: {...}, bucket: {...}, putItemResult: {SdkHttpMetadata: {...}, ...}, pollyResult: {...}, updateItemResult: {...}}`. For the failure path, `$` contains `{..., error: {Error: "...", Cause: "..."}}` — the Cause field often contains full exception stack traces with internal ARNs and table names.

## Test specificity gaps

Three tests scan the entire template JSON as a raw string:

```java
assertTrue(templateJson.contains("sns:Publish"), ...);
assertTrue(templateJson.contains("Catch"), ...);
assertTrue(templateJson.contains("FAILED") && templateJson.contains("errorInfo"), ...);
```

"Catch" could appear in IAM policy conditions, resource metadata, or CDK-generated constructs. "sns:Publish" could match the IAM policy statement rather than the state machine definition (which is arguably also valid, but the test message claims it's checking the state machine). A better approach would extract the `DefinitionString` from the `AWS::StepFunctions::StateMachine` resource and parse it, or use `hasResourceProperties` with `Match` utilities scoped to the state machine.

## Failure path diagnostic gap

The failure DynamoDB update references `$.error.Error` for the `errorInfo` field:

```java
":errorInfo", Map.of("S", JsonPath.stringAt("$.error.Error"))
```

Step Functions Catch places `{"Error": "ErrorType", "Cause": "detailed message"}` at `$.error`. The `$.error.Error` reference resolves only to the error type string (e.g., `"States.TaskFailed"`), discarding the detailed cause. Operators troubleshooting a failure would see "States.TaskFailed" in DynamoDB but not why it failed. Store `$.error.Cause` (or concatenate both) for actionable diagnostics.

## Dead `failurePath` variable

The `failurePath` Chain variable is constructed to wire `failureUpdateTask.next(failurePublishTask)` via CDK's mutating chain API, but the variable itself is never referenced afterward. The wiring works correctly through side effects, but the unused variable reads as dead code. A comment or renaming would clarify intent.
