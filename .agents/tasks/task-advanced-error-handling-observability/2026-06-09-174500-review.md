# Granular error handling, X-Ray tracing, CloudWatch alarms and dashboard

This change layers observability and finer-grained error handling onto the existing Step Functions pipeline. The Polly task gains a retry policy with exponential backoff, both the Lambda and Polly tasks get service-specific Catch blocks ahead of the existing `States.ALL` fallback, X-Ray active tracing is enabled on both the Lambda function and the state machine, and two CloudWatch Alarms plus a dashboard are provisioned. The Lambda handler switches from plain-text logging to structured JSON with request ID correlation. All 89 tests pass (74 existing + 15 new).

**Watch for:** The CloudWatch Alarms have no alarm actions configured (confirmed) — they will change state but not notify anyone. The granular Catch blocks route to the same target with the same resultPath as the fallback, providing no differentiated failure handling today (confirmed). The structured logging in the Lambda handler emits the full incoming event including S3 object keys, which may contain PII in filenames (possible).

## High-level view

Polly gets a dedicated retry with 2-second initial interval and 2x backoff; Lambda gains `Lambda.AWSLambdaException` in its existing retry list. Both tasks now have service-specific Catch blocks evaluated before `States.ALL`, but both the granular and fallback catches route to the identical failure path with the identical resultPath — no differentiated handling exists today.

X-Ray tracing is enabled on both Lambda and the state machine (CDK handles the IAM policy automatically). The two CloudWatch Alarms fire on threshold >= 1 failure within 5 minutes with `treatMissingData: NOT_BREACHING`, but they have no alarm actions attached — they change state in CloudWatch without notifying anyone. The dashboard has two graph widgets but is missing the Lambda Duration metric that ARCHITECTURE.md claims is present.

The Lambda logging refactor swaps to structured JSON with `request_id` correlation but logs the entire incoming event payload verbatim, and there are no unit tests covering the handler changes.

## Alarms without actions

Neither alarm has an alarm action:

```java
Alarm stateMachineFailureAlarm = Alarm.Builder.create(this, "StateMachineExecutionFailedAlarm")
        .metric(stateMachine.metricFailed(...))
        .threshold(1)
        // ... no .actionsEnabled() or addAlarmAction()
        .build();
```

Without an action, the alarm is a dashboard indicator only. It will not page, email, or trigger a Lambda. The `failedTopic` SNS topic already exists and could serve as the target — you can see failures on the dashboard, but you won't know about them unless you're already looking.

## Granular catches: structure without differentiation

Both the granular catch and the `States.ALL` catch route to the same `failureUpdateTask` with the same `resultPath("$.error")`:

```java
// Granular
lambdaInvokeTask.addCatch(failureUpdateTask, CatchProps.builder()
        .errors(List.of("Lambda.ServiceException", "Lambda.AWSLambdaException"))
        .resultPath("$.error")
        .build());
// Fallback
lambdaInvokeTask.addCatch(failureUpdateTask, CatchProps.builder()
        .errors(List.of("States.ALL"))
        .resultPath("$.error")
        .build());
```

The Step Functions `$.error.Error` field already carries the original error type regardless of which Catch block fires, so downstream tasks already get the specific error name. The granular blocks add zero runtime differentiation today. This is defensible as a foundation for future extension (routing service errors to a different remediation path), but as shipped the complexity doesn't buy anything.

## Structured logging: event exposure

The `RECEIVED` log entry serializes the entire Step Functions input:

```python
_log_structured("INFO", request_id, "RECEIVED", "Received event", event=event)
```

This includes `$.bucket.name`, `$.object.key`, and any upstream state data (like `$.putItemResult`). In production with user-uploaded audio files, S3 keys might contain user-identifiable information (account IDs, email-based folder paths). Logging selectively (just bucket and key) would reduce the exposure without losing debuggability.

## Dashboard documentation mismatch

ARCHITECTURE.md states the Lambda widget shows "Invocations, Errors, and Duration" but the implementation only adds two metrics:

```java
dashboard.addWidgets(GraphWidget.Builder.create()
        .title("Lambda Function Metrics")
        .left(List.of(
                audioProcessorFunction.metricInvocations(),
                audioProcessorFunction.metricErrors()
        ))
        .width(12)
        .build());
```

`metricDuration()` is absent from the code. (confirmed)

## Lambda handler test gap

The structured logging change is the one runtime-behavior modification in this diff. Python tests aren't part of the Maven lifecycle, which explains the omission, but the handler is flying uncovered. A minimal pytest file verifying that `_log_structured` produces valid JSON and that `handler` returns the expected schema would catch regressions from future refactors.
