## Diagram arrow semantics

Throughout the diagram, arrows from Lambdas to storage resources represent data writes:

```
PollyTTS -->|"Stores Processed Audio"| S3Output
MetadataExtract -->|"Writes Metadata"| DDB
MetadataExtract -->|"Publishes Notification"| SNSTopic
```

All of these follow "actor тЖТ destination" semantics. But `MetadataExtract -->|"Reads Processed Audio"| S3Output` uses the same visual form to represent a read тАФ the arrow points at the thing being read, not the thing receiving data. An implementor glancing at the diagram sees MetadataExtract "doing something to" S3Output, which looks like a write. In a document that's now the authoritative implementation reference, diagram arrows should be unambiguous without reading labels.

## bedrockEnabled=false edge case

The Feature Flags section states: "The Choice state in the diagram routes between Polly TTS and Bedrock Enhancement based on input type (text script present vs not); this routing logic is only present when `bedrockEnabled` is true."

When `bedrockEnabled=false`:
- No Choice state exists
- No Bedrock Lambda exists
- Presumably, Validate flows directly to Polly TTS

But Polly TTS requires a text script (step 7: "generate soothing voice audio from provided text scripts"). If a user uploads raw audio without a text script in a `bedrockEnabled=false` environment, the document doesn't describe what happens. Possibilities include: validation rejects it (but step 5 doesn't mention this check), or the upload is processed as a no-op, or it's simply not a supported use case in dev environments.

This matters because the document is the "single source of truth" and CDK implementors need to know whether to add a validation check for text-script-required when Bedrock is disabled.
