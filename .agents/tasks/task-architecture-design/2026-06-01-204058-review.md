# Architecture document rewrite for sleep audio pipeline

The original ARCHITECTURE.md was a lightweight sketch: five components, a linear flow, and a simple Mermaid diagram. This change replaces it entirely with a comprehensive, production-grade design covering Step Functions orchestration with Polly/Bedrock processing Lambdas, detailed security posture (IAM per-Lambda, KMS encryption, private buckets), observability (CloudWatch, X-Ray), cost drivers, multi-environment configuration, and a feature-flag strategy. A supporting change to AGENT_GUIDELINES.md promotes this document to "single source of truth" for future implementation work.

**Watch for:**
- The diagram's Choice state implies mutual exclusion between Polly and Bedrock paths, but the prose describes parallel execution — an inconsistency that will mislead implementors (confirmed).
- The Metadata Extraction Lambda is documented with read access to the Output Bucket, but the data flow has it *writing* to the Output Bucket — role/flow mismatch (confirmed).
- No DLQ or dead-letter configuration is described anywhere in the data flow or Step Functions design, despite DLQ alarms being referenced in Observability (confirmed).

---

## High-level view

The document graduates from a placeholder sketch to a full architectural specification, adding nine new sections (Data Flow, AWS Services, Security, Observability, Cost, Multi-Environment, Feature Flags, Future Extensibility, Implementation Status) and expanding the Mermaid diagram from a flat 5-node graph to a multi-subgraph pipeline with 15+ nodes and three connection layers (data flow, observability, security). The architecture now commits to Step Functions as the orchestrator, replacing the hedged "Lambda / Step Functions" framing.

The security section prescribes per-Lambda IAM roles scoped to individual services and SSE-KMS on all storage resources. However, the role definitions contain a conflict between the textual data flow (where Metadata Extraction writes processed audio *and* metadata) and the IAM section (which grants it only read access to the Output Bucket). Implementors will need to reconcile this before coding the CDK role grants.

The multi-environment strategy introduces a feature-flag layer (bedrockEnabled, pollyVoiceType, alarmActionsEnabled) but omits how values propagate from CDK context into the Step Functions state machine definition — whether the Bedrock state is conditionally included at synthesis time or at runtime via Choice state evaluation.

The AGENT_GUIDELINES addition establishes a precedence rule: if implementation conflicts with architecture, the document wins and must be updated first. This is a healthy governance pattern but raises the stakes for accuracy in the architecture doc itself — any inconsistency now blocks implementations rather than being silently overridden.

## Parallel-vs-exclusive Choice in Step Functions

The Data Flow section states in step 6: "Both paths can execute in parallel when applicable." The Mermaid diagram renders this as a diamond-shaped Choice node with two labeled outbound edges ("Text Script Provided" and "AI Enhancement Enabled"), each leading to a different Lambda. In Step Functions, a Choice state evaluates conditions and routes to exactly one branch. Parallel execution requires a Parallel state (or a Map state). These produce fundamentally different CDK constructs — `Choice` with `.when()` conditions vs `Parallel` with `.branch()` definitions — and different IAM, error-handling, and timeout semantics.

The convergence point also matters: both Polly and Bedrock edges currently feed into MetadataExtract, but if they can run in parallel, a join is needed before MetadataExtract receives both outputs. The diagram as drawn doesn't show what happens when *both* conditions are true simultaneously.

## Metadata Extraction role contradicts data flow

The Security section defines Metadata Extraction Lambda Role as: "Read access to the S3 Output Bucket, write access to the DynamoDB Table, and publish permission to the SNS Topic." But the Data Flow assigns three responsibilities to this Lambda: extract metadata (read from Output Bucket), store processed audio (write to Output Bucket in step 10), and write metadata (write to DynamoDB in step 11). Either:

- Step 10 ("Store Processed Audio") should be the responsibility of the preceding Lambda (Polly or Bedrock) rather than Metadata Extraction, or
- The IAM role needs `s3:PutObject` on the Output Bucket.

This matters because the CDK role grants will directly encode this decision.

## DLQ gap between observability and architecture

The Observability section references "DLQ message depth alarm for any configured dead-letter queues," and the Error Handling step (step 13) mentions "configured retries" before failure. But the architecture never defines:

- Whether Lambdas within Step Functions have async invocation DLQs (they don't, since Step Functions invokes them synchronously).
- Whether a separate SQS queue catches events that EventBridge fails to deliver.
- Whether Step Functions catch states route to a DLQ.

An implementor following this document would create CloudWatch alarms pointing at queues that don't exist in the architecture. The services table lists no SQS resource.

## Feature flag compile-time vs runtime ambiguity

The Multi-Environment section says `bedrockEnabled` "controls whether the Bedrock Enhancement step is *included* in the Step Functions state machine." That phrasing ("included") implies CDK synth-time: when `bedrockEnabled=false`, the Bedrock Lambda and its Choice branch are not synthesized into CloudFormation at all.

But the Mermaid diagram unconditionally shows BedrockEnhance as a node, and the Data Flow unconditionally describes step 8 with a qualifier "(Lambda, Optional)." The diagram implies runtime gating (Choice state evaluates the flag), not compile-time removal.

These produce different behaviors in production:
- Compile-time: no Lambda deployed, no IAM role, no cost, but changing the flag requires a CDK redeploy.
- Runtime: Lambda deployed but conditionally invoked, minimal cost, flag change is a state machine input update.

The document should pick one and be explicit, since both the CDK code structure and the test assertions depend on it.

## user_id provenance

The Data Flow (step 11) writes `user_id` to DynamoDB. The High-Level Overview lists it as a metadata field. But the upload path is a raw S3 PutObject — there is no authenticated identity attached to the event. The EventBridge event contains the IAM principal ARN of whoever uploaded, but not an application-level user ID.

The Future Extensibility section lists Cognito as a planned addition, which suggests `user_id` is aspirational. If so, the schema should document it as nullable or derived-from-principal until Cognito lands, so early implementations don't hard-code a requirement for a field that has no source.
