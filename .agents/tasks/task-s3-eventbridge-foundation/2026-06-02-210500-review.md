# S3 buckets and EventBridge rule for Sleep Audio Pipeline

The change adds the foundational storage and event-routing layer: two S3 buckets (input with EventBridge notifications, output without) and an EventBridge rule that triggers on Object Created events from the input bucket, routed to a placeholder CloudWatch LogGroup target. Four CDK assertion tests validate the synthesized template. ARCHITECTURE.md gains an Implementation Status section with Mermaid styling to distinguish implemented from planned components.

**Watch for:** The LogGroup has no explicit retention set, so CDK defaults it to 731 days (2 years) with RETAIN deletion policy â€” this will accumulate cost for a resource documented as temporary (confirmed). The test suite does not distinguish which bucket is the input bucket versus the output bucket â€” `testInputBucketExists` passes if *either* bucket matches (confirmed). The EventBridge rule test does not verify the `detail.bucket.name` filter, so a regression removing the bucket-scoping constraint would go undetected (confirmed).

## High-level view

The two buckets share an identical security posture (SSE-S3, versioning, block public access, SSL enforcement) and differ only in whether EventBridge notifications are enabled. The CloudWatch LogGroup target is a placeholder that will be replaced by a Step Functions target in a future iteration; it currently synthesizes with a 731-day retention and RETAIN deletion policy, creating cost accumulation for a resource meant to be transient.

The test suite verifies resource existence and property subsets but does not pin assertions to specific logical IDs or unique properties, meaning tests will continue to pass even if bucket configurations drift or if the EventBridge rule loses its bucket-name filter.

## LogGroup as placeholder target â€” retention and lifecycle gap

The `SleepAudioEventLogGroup` is created with no explicit `retention` or `removalPolicy`:

```java
LogGroup ruleLogGroup = LogGroup.Builder.create(this, "SleepAudioEventLogGroup")
        .build();
```

CDK's default for `LogGroup` without explicit retention is `RetentionDays.TWO_YEARS` (731 days), and the synthesized template confirms `"RetentionInDays": 731` with `"DeletionPolicy": "Retain"`. For a component explicitly documented as a placeholder:

- Logs from matched EventBridge events will accumulate for two years even after the placeholder is replaced.
- Deleting the stack will orphan the LogGroup (RETAIN policy), leaving a cost-accumulating resource in the account.

Setting `retention(RetentionDays.ONE_WEEK)` and `removalPolicy(RemovalPolicy.DESTROY)` would make the placeholder's transient nature explicit in the infrastructure.

## Test specificity â€” input vs output bucket ambiguity

`testInputBucketExists` asserts that *some* S3 bucket in the template has encryption, versioning, and public access block. Since both buckets share these properties, the test would pass even if the input bucket were accidentally removed and only the output bucket remained.

A stronger assertion would combine the property check with something unique to the input bucket â€” for example, asserting that the bucket associated with the `Custom::S3BucketNotifications` resource also has the expected encryption and versioning properties.

`testOutputBucketExists` only checks `resourceCountIs("AWS::S3::Bucket", 2)`. This verifies quantity but nothing about the output bucket's properties. If the output bucket lost its encryption config, this test would still pass.

## EventBridge rule test â€” missing bucket name filter coverage

The test asserts `source` and `detail-type` but not the `detail.bucket.name` filter:

```java
template.hasResourceProperties("AWS::Events::Rule", Map.of(
    "EventPattern", Map.of(
        "source", List.of("aws.s3"),
        "detail-type", List.of("Object Created")
    ),
    "State", "ENABLED"
));
```

The synthesized template confirms the production rule includes `"detail": { "bucket": { "name": [{ "Ref": "SleepAudioInputBucket..." }] } }`. Without this filter, the rule would match Object Created events from *any* S3 bucket in the account. The test doesn't catch a regression if that constraint is accidentally removed.

## Encryption posture â€” test tied to temporary SSE-S3 choice

The Implementation Status table documents the SSE-S3 â†’ CMK migration as planned work. The test hard-codes `"SSEAlgorithm", "AES256"`, with no TODO or comment marking it as temporary. When the CMK migration lands, this assertion will break silently rather than guiding the developer to update the security verification. A comment in the test noting the future migration would reduce confusion.
